## Phase 1 — Code fixes (commit before pushing)

### 1.1 PostgreSQL SSL — the part that will bite hardest

⚠️ Your audit suggested appending `?sslmode=require` to `DATABASE_URL`. That is **not** enough on DO and can actually *cause* the failure: DO's connection string already carries `sslmode=require`, and `pg` will override your `ssl` object with it, throwing `SELF_SIGNED_CERT_IN_CHAIN`. The reliable fix is to pass DO's CA certificate explicitly.

When you attach a Managed PG database to the app (Phase 4), DO injects `DATABASE_URL` and a CA cert. Bind the cert to `DATABASE_CA_CERT` and read it:

**`lib/db/src/index.ts`**
```ts
import { Pool } from "pg";
import { drizzle } from "drizzle-orm/node-postgres";
import * as schema from "./schema"; // adjust to your actual schema import

const ca = process.env.DATABASE_CA_CERT;

export const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  ssl: ca
    ? { rejectUnauthorized: true, ca }          // production (DO)
    : { rejectUnauthorized: false },            // local/dev fallback
});

export const db = drizzle(pool, { schema });
```

**`artifacts/api-server/src/app.ts`** — the session store currently opens its *own* connection via `conString`, which bypasses the SSL config above and will fail on DO. Reuse the pool instead:
```ts
import { pool } from "@workspace/db"; // the SSL-configured pool from lib/db

const sessionStore = new PgStore({
  pool,                          // ← was: conString: databaseUrl
  tableName: "user_sessions",
  createTableIfMissing: true,
  pruneSessionInterval: 60 * 60,
});
```

**Fallback** if you *still* see `SELF_SIGNED_CERT_IN_CHAIN`: the URL's `sslmode` is winning. Either strip `sslmode` from `DATABASE_URL`, or connect with discrete params (`${db.HOSTNAME}`, `${db.PORT}`, `${db.USERNAME}`, `${db.PASSWORD}`, `${db.DATABASE}`) plus the same `ssl: { rejectUnauthorized: true, ca }` block.

### 1.2 GCS auth — replace the Replit sidecar

The sidecar at `127.0.0.1:1106` does not exist on DO. Create a GCP service account with Storage Object Admin on your bucket, download its JSON key, then base64-encode it (avoids newline mangling in env vars):

```bash
base64 -w0 service-account.json   # Linux
base64 -i service-account.json    # macOS
```

Replace **only the client-init block** in `artifacts/api-server/src/lib/gcsStorage.ts` (leave `uploadToGcs()` and the rest unchanged):
```ts
import { Storage } from "@google-cloud/storage";

function loadCredentials() {
  const b64 = process.env.GCP_SA_KEY_B64;
  if (!b64) throw new Error("GCP_SA_KEY_B64 is required.");
  return JSON.parse(Buffer.from(b64, "base64").toString("utf8"));
}

const credentials = loadCredentials();

const gcsClient = new Storage({
  projectId: credentials.project_id,
  credentials,
});
```

Set `DEFAULT_OBJECT_STORAGE_BUCKET_ID` (Phase 4) to your real GCS bucket name.

### 1.3 Add the Dockerfile

Place at repo root as `Dockerfile`. This is corrected from your audit draft — the prod stage must copy **built** artifacts with `--from=build` (the original copied un-built source from the build context), and the frontend output is under `dist/public`, not `public`:

```dockerfile
# syntax=docker/dockerfile:1
FROM node:24-slim AS base
ENV PNPM_HOME="/pnpm"
ENV PATH="$PNPM_HOME:$PATH"
RUN corepack enable && corepack prepare pnpm@10.26.1 --activate
WORKDIR /app

# ── Build ──────────────────────────────────────────────
FROM base AS build
COPY . .
RUN pnpm install --frozen-lockfile
RUN pnpm --filter @workspace/api-server run build
RUN pnpm --filter @workspace/esign-app run build
# Pruned, production-only deployable of the API (includes its built dist + prod deps)
RUN pnpm --filter @workspace/api-server deploy --prod /prod/api

# ── Production ─────────────────────────────────────────
FROM base AS production
WORKDIR /app
ENV NODE_ENV=production
ENV PORT=8080
COPY --from=build /prod/api ./
# Built frontend served as static files by the API (see 1.4)
COPY --from=build /app/artifacts/esign-app/dist/public ./public
EXPOSE 8080
CMD ["node", "--enable-source-maps", "dist/index.mjs"]
```

> If `pnpm deploy` complains in pnpm 10, add `--legacy` to that line. Path note: this assumes the bundle lands at `/app/dist/index.mjs` and the server reads static files from `../public` relative to `dist/`. **See 1.4 — this is the one thing I need your files to finalize.**

### 1.4 ⬅️ Confirm: does the API serve the frontend?

The single-service shape assumes `app.ts` serves the `public/` directory with SPA fallbacks for `/sign/*` (→ `sign.html`) and `/*` (→ `index.html`). Your audit said the frontend "is not a Node process," so this middleware may **not** exist yet. If it doesn't, it's ~15 lines:

```ts
import express from "express";
import path from "path";
import { fileURLToPath } from "url";

const publicDir = path.join(path.dirname(fileURLToPath(import.meta.url)), "../public");

app.use(express.static(publicDir));
// Specific SPA entry for signing pages
app.get("/sign/*", (_req, res) => res.sendFile(path.join(publicDir, "sign.html")));
// Catch-all SPA entry (register AFTER all /api routes)
app.get("*", (_req, res) => res.sendFile(path.join(publicDir, "index.html")));
```

**Send me `artifacts/api-server/src/app.ts` + `index.ts` + the two build scripts** and I'll hand back the exact Dockerfile paths + this middleware wired correctly. Until confirmed, treat 1.3/1.4 as provisional.

## Phase 4 — Create the App

**Console path:** Create → **Apps** → **GitHub** → select your WorkflowSign repo → branch `main` → **Autodeploy on**. App Platform auto-detects the `Dockerfile`.

Then configure:
1. **Resource type:** Web Service. **HTTP port: `8080`**.
2. **Region:** FRA1 (match the DB).
3. **Attach the database:** add the Phase 3 cluster to this app (or create a dev DB inline). This is what makes the `${db.*}` bindings resolve.
4. **Environment variables** (mark secrets as encrypted):

| Key | Value | Type |
|---|---|---|
| `NODE_ENV` | `production` | plain |
| `PORT` | `8080` | plain |
| `DATABASE_URL` | `${db.DATABASE_URL}` | secret |
| `DATABASE_CA_CERT` | `${db.CA_CERT}` | plain |
| `SESSION_SECRET` | (generate: `openssl rand -hex 32`) | secret |
| `APP_URL` | `${APP_URL}` | plain |
| `APP_ORIGIN` | `${APP_URL}` | plain |
| `GCP_SA_KEY_B64` | (your base64 SA key) | secret |
| `DEFAULT_OBJECT_STORAGE_BUCKET_ID` | your bucket name | plain |
| `SMTP_HOST` / `SMTP_PORT` / `SMTP_USER` / `SMTP_PASS` | if using email | secret |
| `AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` | if using SSO | secret |

`${db.*}` and `${APP_URL}` are App Platform binding variables — they resolve at deploy time. Replace `db` with your DB component's actual name if different.

## Phase 5 — Create the schema in the new database

Run Drizzle against the DO DB **from your laptop** using the public connection string + CA cert (add your IP to Trusted Sources first):

```bash
export DATABASE_URL='postgresql://doadmin:...@<host>:25060/defaultdb?sslmode=require'
export DATABASE_CA_CERT="$(cat ca-certificate.crt)"
# Fresh DB → push the schema directly:
pnpm --filter @workspace/db exec drizzle-kit push
# Or, if you maintain migration files:
pnpm --filter @workspace/db exec drizzle-kit migrate
```

`user_sessions` will auto-create on first run (`createTableIfMissing: true`), but your app tables (documents, document_events, users, …) come from this step. Verify with `\dt` over `psql`.

## Phase 6 — Migrate existing data *(only if preserving production data)*

Skip this entirely if you're starting fresh. If you want existing users + documents + the 83 PDFs to carry over:

**6a. Database rows** — dump from Replit, restore to DO:
```bash
pg_dump --no-owner --no-privileges "$REPLIT_DATABASE_URL" -Fc -f workflowsign.dump
pg_restore --no-owner --no-privileges \
  -d "postgresql://doadmin:...@<host>:25060/defaultdb?sslmode=require" \
  workflowsign.dump
```
Run this *after* Phase 5 if the dump is data-only, or *instead of* `drizzle-kit push` if the dump includes schema. Don't do both for the same tables.

**6b. The 83 PDFs** — they're in `bucket-export/`. Upload to your new GCS bucket, preserving the object keys so the existing `gcs://bucket/key` rows still resolve:
```bash
gcloud auth activate-service-account --key-file=service-account.json
gsutil -m cp -r bucket-export/* gs://your-gcs-bucket/
```
If the bucket name in the keys differs from the new bucket, update `documents.filepath` rows accordingly.
