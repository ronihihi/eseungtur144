# WorkflowSign — Post-Fix Review & Report (Replit Agent Prompt, READ-ONLY)

> **How to use:** paste everything below the line into the Replit agent. Run this *after* the recommended fixes have been applied. The agent only reviews and writes a report — it must not change the app.

---

You are a senior security engineer doing a **read-only audit** of **WorkflowSign** (Node.js + Express + TypeScript, React/Vite frontend, Drizzle ORM on PostgreSQL, object storage for PDFs, Azure SSO + local auth, nodemailer). A set of security and bug fixes was recently applied. Your job is to **verify the codebase, confirm the fixes are correctly in place, check for any errors, and find anything still broken or newly broken — then write a report.**

## Hard rule

**Do not modify, create, or delete any application code. No edits, no commits, no branches, no dependency changes, no auto-formatting.** The only file you may write is `REVIEW-REPORT.md` at the repo root. If you believe a change is needed, describe it in the report — do not make it.

## What to do

1. **Verify the prior fixes.** For each item in the checklist below, find the relevant code and decide its status: ✅ Fixed · ⚠️ Partial · ❌ Not done · 🔁 Regressed (newly broken). Record the exact `file:line` as evidence. If `WorkflowSign-Security-Remediation.md` is in the repo, use it as the spec for what each fix should look like.

2. **Check for errors.** Run and capture the output of each (use whatever the repo defines in `package.json`):
   - typecheck — `tsc --noEmit` (or the project's typecheck script)
   - production build — the build script for backend and frontend
   - lint — the lint script, if present
   - tests — the test script, if present
   - boot — start the app/dev server and note any startup/runtime errors (stop it after it boots)
   Report every error and meaningful warning verbatim, with the file it points to.

3. **Grep for regressions / dangerous patterns** and report any hits:
   - `rejectUnauthorized: false`, `NODE_TLS_REJECT_UNAUTHORIZED`
   - `x-forwarded-for` used directly for a stored IP (should be `req.ip`)
   - raw template interpolation of user input in email HTML
   - any `/documents/:id*` route missing the `uploadedBy = session.userId` scope
   - secrets or tokens hardcoded in source

4. **Fresh sweep.** Independently look for any remaining or new bugs, security issues, or logic errors anywhere in the app — including the spots that may not have been fully reviewed before: the public **`/sign/:token/file`** download route, the **PDF signer** internals, and the **frontend pages**.

## Checklist — confirm each is in place

Severity: **P0** security · **P1** bug/hardening · **P2** cleanup.

| ID | Confirm this is in place |
|----|--------------------------|
